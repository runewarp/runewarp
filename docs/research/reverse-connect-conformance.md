# Reverse CONNECT conformance and implementation viability

Research for [Research Reverse CONNECT conformance and implementation viability](https://github.com/runewarp/runewarp/issues/168), current on 15 July 2026.

> Planning research only. This records evidence and does not specify Runewarp's implemented protocol.

## Answer

Reverse HTTP CONNECT revision 01 is viable for an isolated, draft-pinned prototype over Runewarp's existing Quinn transport, using hyperium's `h3` and a small streaming Capsule codec. It is not ready to define a stable Runewarp wire protocol or claim third-party interoperability. The Reverse CONNECT control Capsule types are still `TBD`; `connect-listen`, `connect-accept`, and their default URI suffixes are requested but unregistered; and dependent CONNECT-TCP revision 11 still uses provisional test identifiers for its TCP data capsules. The most direct Rust stack, `h3` + `h3-quinn`, explicitly describes itself as experimental despite active releases and interoperability testing. [Reverse CONNECT -01, Sections 7.1–7.3](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-7) [CONNECT-TCP -11, Sections 8.1–8.3](https://www.ietf.org/archive/id/draft-ietf-httpbis-connect-tcp-11.html#section-8) [IANA HTTP Upgrade Tokens](https://www.iana.org/assignments/http-upgrade-tokens/http-upgrade-tokens.xhtml) [IANA MASQUE registries](https://www.iana.org/assignments/masque/masque.xhtml) [hyperium/h3 status](https://github.com/hyperium/h3#status)

Recommendation: keep HTTP/3 Reverse CONNECT evidence-gated. If prototyped, pin the exact draft revisions and provisional identifiers behind one replaceable codec boundary, revalidate both drafts and IANA registries at pickup, and measure setup latency and small-write framing before considering a protocol decision.

## Standards status and churn

`draft-rosomakho-masque-reverse-connect-01` was published in the IETF archive on 21 June 2026 as an author Internet-Draft with a Standards Track intent and December 2026 expiry; its official source tag resolves to commit [`6a4d3c873c15d9badbdf9042d96a35eac6f6814e`](https://github.com/yaroslavros/draft-masque-reverse-connect/tree/6a4d3c873c15d9badbdf9042d96a35eac6f6814e). The Datatracker status page still exposes only expired revision 00, so the archive and tagged source are the evidence for revision 01 while Datatracker's adoption/status view is stale. Internet-Drafts are works in progress that can be replaced or obsoleted. [Reverse CONNECT -01 archive header and boilerplate](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#name-status-of-this-memo) [stale Datatracker record](https://datatracker.ietf.org/doc/draft-rosomakho-masque-reverse-connect/) [official source repository and release](https://github.com/yaroslavros/draft-masque-reverse-connect)

Revision 01 changed no protocol text from revision 00: an exact comparison of the official text changes only publication metadata and updates the normative CONNECT-TCP reference from revision 07 to revision 11. The Reverse CONNECT surface therefore has little demonstrated revision churn of its own, but it imports an independently moving draft. [Reverse CONNECT -00 references](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-00.html#section-8) [Reverse CONNECT -01 references](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-8)

The imported `draft-ietf-httpbis-connect-tcp-11` is an active HTTPBIS working-group draft published 20 March 2026. Its permanent `connect-tcp`, DATA, and FINAL_DATA registrations are not assigned. Revision 11 says interoperability tests must use Upgrade token `connect-tcp-07` and provisional Capsule values `0x2028d7f0` (`DATA-08`) and `0x2028d7f1` (`FINAL_DATA-08`). Those revision-suffixed names are direct evidence that a prototype must isolate draft-version choices rather than expose them as Runewarp canon. [CONNECT-TCP -11 status](https://datatracker.ietf.org/doc/html/draft-ietf-httpbis-connect-tcp-11) [CONNECT-TCP -11, Section 8](https://www.ietf.org/archive/id/draft-ietf-httpbis-connect-tcp-11.html#section-8)

Reverse CONNECT itself requests, only "if approved", the Upgrade tokens `connect-listen` and `connect-accept`, MASQUE suffixes `listen` and `accept`, and permanent Capsule types AVAILABLE_SERVICES, CONNECTION_REQUEST, and CONNECTION_REQUEST_DECLINED. All three Capsule values remain `TBD`. The current IANA registries contain none of those entries and also contain neither CONNECT-TCP data Capsule. [Reverse CONNECT -01, Section 7](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-7) [IANA HTTP Upgrade Tokens](https://www.iana.org/assignments/http-upgrade-tokens/http-upgrade-tokens.xhtml) [IANA MASQUE URI Suffixes and HTTP Capsule Types](https://www.iana.org/assignments/masque/masque.xhtml)

## Inherited HTTP/3 contract

For HTTP/3, a conforming Reverse CONNECT implementation has this shape:

1. Configure two absolute URI Templates: one for the Listener Control Channel and one for accepting individual connection requests. The listener template can scope `target` and `ipproto`; the acceptance template must contain `request_id` and meet the draft's level, component, ASCII, and expansion restrictions. [Reverse CONNECT -01, Sections 3, 4.1, and 5.2](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-3)
2. Establish the long-lived listener as Extended CONNECT with `:method = CONNECT`, `:protocol = connect-listen`, the derived `:scheme`, `:authority`, and `:path`, and `capsule-protocol = ?1`; success is a 2xx response using the Capsule Protocol. This imports CONNECT-IP's HTTP/3 requirements, including server advertisement of `SETTINGS_ENABLE_CONNECT_PROTOCOL`. The setting is registered as `0x08`, defaults to disabled, and an unsupported Extended CONNECT protocol should receive 501. [Reverse CONNECT -01, Section 4.3](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-4.3) [RFC 9484, Section 4.4](https://www.rfc-editor.org/rfc/rfc9484.html#section-4.4) [RFC 9220, Section 3](https://www.rfc-editor.org/rfc/rfc9220.html#section-3)
3. For each inbound session, the proxy sends CONNECTION_REQUEST on the listener, containing a listener-unique Request ID and Service. The client either sends CONNECTION_REQUEST_DECLINED or opens a new request stream on the same HTTP/3 connection using Extended CONNECT with `:protocol = connect-accept` and a URI derived from that Request ID. A malformed request should receive 400; an unmatched Request ID should receive 404; acceptance requires 2xx plus the Capsule Protocol, otherwise the client must abort the connection attempt. [Reverse CONNECT -01, Sections 5.1 and 5.2.2](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-5)
4. Only after the client receives the successful acceptance response does the draft direct it to establish the requested local or internal TCP socket. Local-connect failure requires immediate closure of the request stream. TCP bytes are the ordered concatenation of CONNECT-TCP DATA and FINAL_DATA Capsule payloads; FIN maps to FINAL_DATA, and abnormal HTTP/3 closure uses `H3_CONNECT_ERROR`. [Reverse CONNECT -01, Section 5.3](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-5.3) [CONNECT-TCP -11, Sections 3 and 3.4](https://www.ietf.org/archive/id/draft-ietf-httpbis-connect-tcp-11.html#section-3)
5. The proxy must restrict Reverse CONNECT to authenticated users and enforce policy over advertised Services. The client must authenticate the proxy and restrict inbound sessions to intended destinations. AVAILABLE_SERVICES is only a replace-all hint: omission does not prevent the proxy from requesting an unadvertised Service, and inclusion does not promise health. [Reverse CONNECT -01, Sections 4.4 and 6](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-4.4)

The Capsule Protocol adds strict parser duties: unknown Capsule types must be skipped, malformed or truncated Capsules make the HTTP message malformed, prohibited content headers and status codes must be rejected, and parsers should stream values instead of buffering a whole advertised length because doing so can consume flow-control credit and deadlock. [RFC 9297, Sections 3.2–3.4](https://www.rfc-editor.org/rfc/rfc9297.html#section-3.2)

## Latency and performance risks

On an already established listener, the specified visitor setup adds approximately one full Client–Server network RTT before local connection setup compared with Runewarp's current Server-opened QUIC stream. This is an inference from the normative sequence: CONNECTION_REQUEST travels Server to Client, acceptance HEADERS travel Client to Server, the success response travels Server to Client, and only then does the Client start its local TCP connection. The current protocol can deliver the tunneled ClientHello in the first Server-to-Client stream flight. Reverse CONNECT therefore needs bounded buffering or backpressure for the accepted Visitor socket during the added exchange. The draft defines no optimistic acceptance or payload fast path. [Reverse CONNECT -01, Sections 5.1–5.3](https://www.ietf.org/archive/id/draft-rosomakho-masque-reverse-connect-01.html#section-5) [current Runewarp Tunnel protocol](../protocol.md)

A cold path also pays QUIC/TLS establishment, HTTP/3 control-stream and SETTINGS exchange, and listener request/response setup. HTTP/3 permits 0-RTT only with remembered compatible settings and replay-aware application policy, so resumed early data is not a safe baseline for this authenticated control surface. A persistent Tunnel connection amortizes that cold cost, but reconnect behavior still needs measurement. [RFC 9114, Sections 7.2.4.2 and 10.9](https://www.rfc-editor.org/rfc/rfc9114.html#section-7.2.4.2)

TCP payload is double-framed: every CONNECT-TCP DATA or FINAL_DATA Capsule has variable-length type and length fields, and those bytes are carried inside HTTP/3 DATA frames with another variable-length type and length, atop QUIC STREAM frames. Large coalesced chunks make the fixed framing small; mapping tiny socket reads one-to-one creates avoidable bandwidth, syscall, allocation, and scheduling costs. CONNECT-TCP explicitly permits intermediaries to merge or split adjacent data Capsules while preserving content order and finality, so batching is conformant but trades latency against efficiency and must be benchmarked. [RFC 9297, Sections 3.1–3.2](https://www.rfc-editor.org/rfc/rfc9297.html#section-3.1) [CONNECT-TCP -11, Section 3](https://www.ietf.org/archive/id/draft-ietf-httpbis-connect-tcp-11.html#section-3)

QUIC avoids transport head-of-line blocking between different request streams, but each tunneled TCP flow remains reliable and ordered, so loss blocks that flow. All flows still share connection congestion control and connection-level flow control, while each request stream also has stream flow control. Request and response HEADERS add QPACK work per Visitor; QPACK lets an encoder trade compression against blocking, so a prototype should start with conservative dynamic-table use and measure header and allocation costs. [RFC 9000, Sections 2.2, 4, and 7](https://www.rfc-editor.org/rfc/rfc9000.html#section-2.2) [RFC 9114, Section 4.2.1](https://www.rfc-editor.org/rfc/rfc9114.html#section-4.2.1)

## Maintained Rust implementation viability

### Preferred prototype seam: `h3` + `h3-quinn`

Runewarp currently uses Quinn 0.11.11, Tokio, and rustls 0.23. `h3-quinn` 0.0.10 depends on Quinn 0.11.7-compatible releases, `h3` 0.0.8, and Tokio, so it fits the present transport ecosystem without replacing the QUIC stack. It exposes a Quinn-backed HTTP/3 connection; the `h3` client and server builders both advertise Extended CONNECT, and arbitrary `:protocol` values can be parsed into `h3::ext::Protocol`. [Runewarp dependencies](../../Cargo.toml) [`h3-quinn` 0.0.10 dependencies and API](https://docs.rs/h3-quinn/latest/h3_quinn/) [`h3` client Builder](https://docs.rs/h3/latest/h3/client/struct.Builder.html#method.enable_extended_connect) [`h3` server Builder](https://docs.rs/h3/latest/h3/server/struct.Builder.html#method.enable_extended_connect) [`h3::ext::Protocol`](https://docs.rs/h3/latest/h3/ext/struct.Protocol.html)

The remaining primitives are present: the client request sender is cloneable for concurrent Visitor streams; client and server RequestStreams send and receive body bytes, expose reset controls, and split into independently driven send/receive halves. That is sufficient to layer an incremental Capsule codec and bridge each accepted stream to a TCP half-duplex pair. `h3` does not expose a general Capsule codec in its public module set, so Reverse CONNECT's control and CONNECT-TCP data Capsules remain Runewarp work. [`h3::client::SendRequest`](https://docs.rs/h3/latest/h3/client/struct.SendRequest.html) [`h3` client RequestStream](https://docs.rs/h3/latest/h3/client/struct.RequestStream.html) [`h3` server RequestStream](https://docs.rs/h3/latest/h3/server/struct.RequestStream.html) [`h3` public modules](https://docs.rs/h3/latest/h3/)

The blocker is maturity, not primitive availability. The project says `h3` is "still very experimental", may contain bugs, may change its API, and should be used at the operator's risk. It does maintain HTTP/3 conformance tracking and tests Quinn, s2n-quic, and MsQuic adapters in the QUIC interop runner. That supports a prototype and comparative benchmark, not an unqualified production dependency decision. [hyperium/h3 status and interoperability](https://github.com/hyperium/h3#status)

### Alternative: `quiche`

Cloudflare's `quiche` 0.29.2 is actively maintained and deployed in Cloudflare's edge. Its Rust HTTP/3 API can advertise Extended CONNECT, inspect the peer setting, and send or receive raw request/response bodies, so it can also host the custom Capsule layer. However, it is a lower-level stack whose application owns packet I/O, timers, and the event loop; adopting it would replace or duplicate Runewarp's existing Quinn transport seam. It is therefore a stronger maturity reference and fallback prototype than the simplest integration path. [`quiche` repository status and integration model](https://github.com/cloudflare/quiche#readme) [`quiche::h3::Config`](https://docs.rs/quiche/latest/quiche/h3/struct.Config.html) [`quiche::h3::Connection`](https://docs.rs/quiche/latest/quiche/h3/struct.Connection.html)

## Decision boundary

The evidence makes one next step sharp: a measured, non-production prototype can test whether the added RTT and layered framing are acceptable using `h3` over Quinn. A stable protocol implementation is not yet sharp because essential control and data codepoints remain provisional or unassigned. Revalidate Reverse CONNECT, CONNECT-TCP, the IANA registries, and `h3` maturity before pickup; do not embed draft identifiers into evergreen configuration or domain vocabulary.
